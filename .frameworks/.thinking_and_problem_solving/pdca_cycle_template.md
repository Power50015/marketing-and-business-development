## Before You Start — Critical Instructions

> **The single most important rule of the PDCA Cycle:**
> _"The PDCA cycle is not a linear project plan; it is a continuous loop. The 'Do' phase is NOT when you roll out the solution to the entire company—it is a small-scale pilot test. You must 'Check' the results of that pilot before you 'Act' to standardize it. If you skip the pilot, you are gambling, not improving."_
> _(Source: ASQ — Continuous Improvement; Lean.org — Problem Solving, 2025)_

**How to use this template:**

**Step 1** — **PLAN:** Identify the root cause of a problem and propose a specific countermeasure.
**Step 2** — **DO:** Run a small, controlled experiment (pilot test) of your countermeasure.
**Step 3** — **CHECK:** Analyze the data from the pilot test. Did it work?
**Step 4** — **ACT:** If it worked, standardize the new process (SOP). If it failed, adjust your hypothesis and start the cycle over.

> ⚠️ **Research warning:** Do not treat symptoms in the "Plan" phase. You must use a Root Cause Analysis tool (like the **5 Whys** or **Fishbone Diagram**) to ensure you are solving the actual disease, not just taking painkillers for the symptom.
> _(Source: Factbird — Root Cause Analysis, 2025)_

> 🔁 **Review cycle:** The cycle is infinite. Once you standardize a process in the "Act" phase, that new standard becomes the baseline for your _next_ PDCA cycle to improve upon.

## 🔴 PHASE 4: ACT (Standardize or Pivot)

_Based on the Check phase, make a final decision._

**Option A: The Pilot SUCCEEDED (Standardize)**

- [ ] Document the new process into the official SOP (Standard Operating Procedure).
- [ ] Train the rest of the company/team on the new standard.
- [ ] **Next Step:** What is the _next_ problem we can solve to improve this further? (Start a new PDCA cycle).

**Option B: The Pilot FAILED or was FLAWED (Pivot)**

- [ ] Acknowledge the failure as a successful learning experiment.
- [ ] Why did it fail? Was the root cause wrong, or was the execution poor?
  >
- [ ] **Next Step:** Return to the PLAN phase with this new knowledge and formulate a new hypothesis.

## When to Use This Framework

| Situation                     | How This Template Helps                                                                            |
| ----------------------------- | -------------------------------------------------------------------------------------------------- |
| Recurring defects or bugs     | Stops teams from applying "band-aids" to the same issue every month by forcing Root Cause Analysis |
| High-risk process changes     | The "Do" (Pilot) phase ensures that a flawed new idea doesn't break the entire company's workflow  |
| Implementing Lean / Six Sigma | PDCA is the engine that drives continuous improvement cultures (Kaizen)                            |

_(Source: Aevo Innovate — PDCA Workflow, 2025; Pearl Academy — Continuous Improvement, 2025)_

---

## Framework Limitations — Know What It Cannot Do

| Limitation                       | What to Use Instead                                                                                                                            |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| Urgent Crisis Management         | PDCA is methodical and slow. If the building is on fire, put it out first. Do the PDCA later to figure out why the fire started.               |
| Radical Innovation               | PDCA is for _incremental_ improvement of existing processes. If you need to invent a completely new product category, use **Design Thinking**. |
| Complex, multi-variable problems | If the problem requires complex statistical analysis, use the **DMAIC framework** (Define, Measure, Analyze, Improve, Control) from Six Sigma. |

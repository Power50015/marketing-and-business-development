## Before You Start — Critical Instructions

> **The single most important rule of the Kano Model:**
> *"Never ask a user 'How important is this feature?' They will tell you everything is important. You must ask a pair of questions for every feature: How would you feel if it is present (Functional), and how would you feel if it is absent (Dysfunctional). The cross-section of these two answers reveals the truth."*

> ⚠️ **Research warning:** 
> Kano categories have an expiration date. What is considered an "Attractive/Delighter" feature today (e.g., Free Wi-Fi in a hotel) becomes a "Must-Be" expectation over time. Re-run this survey annually.

> 🔁 **Review cycle:** 
> Do not test more than 15-20 features in a single survey, otherwise, respondents will suffer from survey fatigue and provide inaccurate data.

## Phase 1: The Kano Questionnaire Setup

For every feature you are evaluating, you must present the following pair of questions using the exact 5-point scale below.

### The Standard 5-Point Response Scale
For both questions below, the user must select one of these five options:
1. I like it
2. I expect it
3. I am neutral
4. I can live with it (or tolerate it)
5. I dislike it

### Feature 1: [Insert Feature Name/Description]
- **Functional Question:** How do you feel if [Product Name] HAS [Feature]?
- **Dysfunctional Question:** How do you feel if [Product Name] DOES NOT HAVE [Feature]?

### Feature 2: [Insert Feature Name/Description]
- **Functional Question:** How do you feel if [Product Name] HAS [Feature]?
- **Dysfunctional Question:** How do you feel if [Product Name] DOES NOT HAVE [Feature]?

*(Repeat for up to 15-20 features)*

## Phase 2: The Evaluation Matrix

Once you have the survey results, cross-reference the Functional Answer with the Dysfunctional Answer using this logic table to categorize the feature.

| Functional Answer ↓ \ Dysfunctional → | 1. Like it | 2. Expect it | 3. Neutral | 4. Live with it | 5. Dislike it |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1. Like it** | Questionable (Q) | Attractive (A) | Attractive (A) | Attractive (A) | Performance (P) |
| **2. Expect it** | Reverse (R) | Indifferent (I) | Indifferent (I) | Indifferent (I) | Must-Be (M) |
| **3. Neutral** | Reverse (R) | Indifferent (I) | Indifferent (I) | Indifferent (I) | Must-Be (M) |
| **4. Live with it** | Reverse (R) | Indifferent (I) | Indifferent (I) | Indifferent (I) | Must-Be (M) |
| **5. Dislike it** | Reverse (R) | Reverse (R) | Reverse (R) | Reverse (R) | Questionable (Q) |

---

## Phase 3: Prioritization Strategy

Based on the categorization above, build your product roadmap in this exact order:

1. **Must-Be (M):** Build these first. If you don't have them, users won't even consider your product. (They don't create satisfaction, but their absence creates anger).
2. **Performance (P):** Build these second. The better you execute these, the happier the customer is. This is where you compete with rivals.
3. **Attractive / Delighters (A):** Add a few of these. These are your "Wow" factors. Users don't expect them, but love them when they see them.
4. **Indifferent (I):** DO NOT BUILD. Users do not care about these features. Building them wastes money.
5. **Reverse (R):** DO NOT BUILD. Users actively dislike these features.

---

## Diagnostic Questions to Test Your Survey

1. **Did you provide visual context?** Did you show a mockup or wireframe of the feature before asking the question? (Text descriptions alone often confuse users).
2. **Did you ask the questions back-to-back?** The functional and dysfunctional questions must be presented together to maintain context.
3. **Is your sample size appropriate?** Do you have at least 15-20 responses from your actual target demographic?

## When to Use This Framework

| Situation | How This Template Helps |
| :--- | :--- |
| **Product Roadmap Planning** | Stops the team from building "bloated" products with features nobody cares about. |
| **MVP Definition** | Helps identify exactly which "Must-Be" features are required for launch. |
| **Feature Deprecation** | Identifies "Indifferent" features that can be removed to save maintenance costs. |

---

## Framework Limitations — Know What It Cannot Do

| Limitation | What to Use Instead |
| :--- | :--- |
| **Does not account for cost/effort** | The Kano model tells you what users want, not how much it costs to build. Combine it with an **Effort vs. Impact Matrix** or **Value-Complexity Matrix**. |
| **Qualitative, not strict ROI** | It measures satisfaction, not direct financial return. Use **Cost-Benefit Analysis** for ROI. |
